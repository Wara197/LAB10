## 📚 เรื่อง 1: Fetch API และ HTTP Requests

1. ฟังก์ชัน `fetch()` ใช้ส่ง HTTP request ประเภทไหนเป็นค่าเริ่มต้น

- ตอบ : fetch() ถ้าไม่ใส่ options จะส่ง HTTP GET request เป็นค่าเริ่มต้น ซึ่งเหมาะกับการดึงข้อมูลจาก API โดยไม่ต้องกำหนด method เอง

## 🎨 เรื่อง 2: SPA และ Hash-based Routing

1. SPA ย่อมาจากคำอะไร และมีข้อดีอย่างไรกว่า Traditional Web

- ตอบ : SPA (Single Page Application) คือเว็บที่โหลดหน้าเดียวแล้วเปลี่ยนเนื้อหาภายในโดยไม่ต้อง reload ทั้งหน้า ข้อดีคือเร็วขึ้น ประสบการณ์ผู้ใช้ลื่นไหล และลดการโหลดซ้ำของ resource

2. Hash-based routing เมื่อเปลี่ยน URL hash จะทำให้ browser reload หน้าหรือไม่

- ตอบ : Hash-based routing เมื่อเปลี่ยนค่า hash (#/...) browser จะไม่ reload หน้าใหม่ แต่ JavaScript จะตรวจจับการเปลี่ยนแปลงและ render เนื้อหาที่ตรงกับ route

3. Route `#/movie/:id` ใช้ตัวแปร `:id` ประเภทอะไร (ยกตัวอย่าง imdbID จริง)

- ตอบ : Route #/movie/:id ใช้ :id เป็น parameter เช่น IMDb ID จริง tt1375666 ซึ่งเป็นตัวระบุเฉพาะของหนังแต่ละเรื่อง

4. ฟังก์ชัน `navigate()` จะถูกเรียกใช้ เมื่อเกิดเหตุการณ์ใดสองครั้ง

- ตอบ : ฟังก์ชัน navigate() จะถูกเรียกเมื่อเกิด สองเหตุการณ์ คือ hashchange (เปลี่ยน URL hash) และ load (โหลดหน้าเว็บครั้งแรก)

5. คำสั่ง `hash.slice(2)` ทำไมถึงต้องตัด 2 ตัวอักษร (ตัดอะไรออก)

- ตอบ : คำสั่ง hash.slice(2) ใช้เพื่อตัด #/ ออกไป เหลือเฉพาะ path ที่ตามหลัง เช่น "movie/tt1375666"

6. ในโค้ด router ถ้า path มีค่าเป็น `#/movie/tt1375666` จะถูก split เป็น parts อะไรบ้าง

- ตอบ : ถ้า path เป็น #/movie/tt1375666 เมื่อ split จะได้ parts = ["movie", "tt1375666"] โดย index 0 คือ page, index 1 คือ parameter

7. โค้ดที่จะแสดงใน `appDiv.innerHTML` มาจาก switch case กี่กรณี และมีกรณีไหนบ้าง

- ตอบ : ใน switch case มี 4 กรณีหลัก คือ "home", "movies", "movie", "search" และมี default สำหรับ route ที่ไม่ตรง → รวมทั้งหมด 5 กรณี

8. ฟังก์ชัน `updateNavigation()` ทำหน้าที่อะไร (ให้อธิบายการเปลี่ยนแปลง CSS class)

- ตอบ : ฟังก์ชัน updateNavigation() ทำหน้าที่ปรับ CSS class ของ <nav a> โดยเพิ่ม class "active" ให้ลิงก์ที่ตรงกับหน้า และลบออกจากลิงก์อื่น เพื่อให้ผู้ใช้เห็นว่าตอนนี้อยู่หน้าไหน

9. ถ้า user ไปที่ route ที่ไม่มีใน switch case จะแสดงอะไร

- ตอบ : ถ้า user ไปที่ route ที่ไม่มีใน switch case จะ render Views.notFound() ซึ่งแสดงข้อความ 404 - ไม่พบหน้า

10. Event listener `hashchange` จะถูก trigger เมื่อ URL เปลี่ยนเป็น `#/movies` หรือไม่

- ตอบ : Event listener hashchange จะ trigger เมื่อ URL เปลี่ยนเป็น #/movies แน่นอน เพราะ hash ส่วนหลังถูกเปลี่ยน ทำให้ router เรียก navigate() อีกครั้ง

## 💾 เรื่อง 3: API Caching และ Performance

1. วัตถุประสงค์ของ `API.cache` object คืออะไร

- ตอบ : วัตถุประสงค์ของ API.cache คือเก็บข้อมูลที่เคยดึงมาแล้ว เพื่อลดการเรียก API ซ้ำ ๆ ทำให้แอปโหลดเร็วขึ้นและลดการใช้ทรัพยากรเครือข่าย

2. ฟังก์ชัน `getMovies()` จะตรวจสอบ cache ก่อนหรือหลังจาก fetch

- ตอบ : ฟังก์ชัน getMovies() จะตรวจสอบ cache ก่อน ถ้ามีข้อมูลอยู่แล้วจะคืนค่าเลย ไม่ต้อง fetch ใหม่จาก API

3. Cache key สำหรับภาพยนตร์ คืออะไร (ดูจาก `this.cache.movies`)

- ตอบ : Cache key สำหรับรายการภาพยนตร์คือ this.cache.movies ซึ่งเก็บผลลัพธ์ของการค้นหาคำว่า "movie" ไว้ใช้ซ้ำ

4. ถ้าภาพยนตร์มีข้อมูลใหม่ออกมา cache เก่าจะลบออกอัตโนมัติหรือไม่

- ตอบ : Cache เก่าไม่ได้ถูกลบอัตโนมัติเมื่อมีข้อมูลใหม่ ต้องเขียนโค้ดเพิ่มเองถ้าต้องการ refresh หรือ clear cache

5. ประโยชน์ของการใช้ cache คืออะไร (ยกเหตุผล 2 ประการ)

- ตอบ : เงื่อนไขในการล้าง cache เช่น
  1.ลดจำนวนการเรียก API ซ้ำ ๆ ทำให้การโหลดข้อมูลเร็วขึ้นและประสบการณ์ผู้ใช้ดีขึ้น
  2.ประหยัด bandwidth และลดภาระการทำงานของ server เพราะไม่ต้องส่ง request ไปทุกครั้ง

6. ปัญหาเมื่อใช้ cache คืออะไร และแก้ไขอย่างไร

- ตอบ : ปัญหาของ cache คืออาจทำให้ข้อมูลไม่ทันสมัย วิธีแก้คือเพิ่มกลไก refresh หรือกำหนดอายุของ cache (cache expiration)

7. Cache ของ movie detail เก็บด้วย key ไหน (imdbID หรือ title)

- ตอบ : Cache ของ movie detail เก็บด้วย key ที่เป็น IMDb ID เช่น tt1375666 เพื่อให้เข้าถึงหนังแต่ละเรื่องได้ตรงและไม่ซ้ำกัน

8. ถ้า user ค้นหา "Inception" สองครั้ง จะใช้ API ทั้งสองครั้งหรือไม่

- ตอบ : ถ้า user ค้นหา "Inception" สองครั้ง ฟังก์ชันจะใช้ cache ที่มีอยู่แล้ว ไม่ต้องเรียก API ซ้ำ ทำให้เร็วขึ้นและประหยัดการเรียก API

9. การรีเซ็ต cache ทำได้อย่างไร (ถ้าต้องการข้อมูลใหม่)

- ตอบ : การรีเฟรช cache ทำได้โดยการลบค่าใน cache หรือเขียนโค้ดให้บังคับ fetch ใหม่ เช่นการกด refresh หรือ clear cache ก่อนเรียก API

10. Cache มีพื้นที่เก็บข้อมูลจำกัดหรือไม่ ควรมี strategy สำหรับการลบหรือไม่

- ตอบ : Cache มีข้อจำกัดเรื่องพื้นที่เก็บข้อมูล จึงควรมี strategy เช่น LRU (Least Recently Used) หรือ manual clear เพื่อจัดการไม่ให้ cache โตเกินไป

## 🔗 เรื่อง 4: CORS และ Cross-Origin Requests

1. CORS ย่อมาจากคำอะไร และเหตุใดจึงมี CORS policy

- ตอบ : CORS (Cross-Origin Resource Sharing) คือกลไกที่อนุญาตให้เว็บจากโดเมนหนึ่งเข้าถึง resource ของอีกโดเมนได้ มี policy เพื่อป้องกันการเข้าถึงข้อมูลโดยไม่ได้รับอนุญาต

2. Browser จะบล็อก request ข้ามโดเมน (cross-origin) ด้วยเหตุผลใด

- ตอบ : Browser จะบล็อก request ข้ามโดเมนเพื่อความปลอดภัย ป้องกันการโจมตีแบบ CSRF หรือการดึงข้อมูลส่วนตัวจากโดเมนอื่นโดยไม่ได้รับอนุญาต

3. OMDB API รองรับ CORS หรือไม่ ยกหลักฐานจากการทำงาน

- ตอบ : OMDB API รองรับ CORS เพราะสามารถเรียกใช้งานผ่าน fetch() ได้โดยตรงจาก client-side และไม่เกิด error เมื่อทดสอบ

4. ความแตกต่างระหว่าง same-origin และ cross-origin request คืออะไร

- ตอบ : Same-origin request คือการเรียกข้อมูลจากโดเมนเดียวกัน ส่วน cross-origin request คือการเรียกข้อมูลจากโดเมนต่างกัน เช่น frontend อยู่ที่ localhost แต่ API อยู่ที่ omdbapi.com

5. เมื่อเกิด CORS error ข้อความจะเป็นแบบไหน (ยกตัวอย่าง)

- ตอบ : เมื่อเกิด CORS error ข้อความที่เห็นใน console มักเป็น "Access to fetch at '...' from origin '...' has been blocked by CORS policy"

6. Server ต้องส่ง HTTP header ไหนเพื่ออนุญาต CORS request

- ตอบ : Server ต้องส่ง HTTP header Access-Control-Allow-Origin: \* หรือระบุ origin ที่อนุญาต เพื่อให้ browser ยอมรับการเรียกข้ามโดเมน

7. `fetch()` API จะส่ง CORS preflight request (OPTIONS) เมื่อไร

- ตอบ : fetch() จะส่ง preflight request (OPTIONS) เมื่อมีการใช้ method ที่ไม่ใช่ GET/POST ธรรมดา หรือมี custom headers เช่น Authorization

8. API key ใน URL ของ OMDB API มีประโยชน์ต่อ CORS policy หรือไม่

- ตอบ : API key ใน URL ของ OMDB ไม่เกี่ยวกับ CORS โดยตรง มันใช้สำหรับการตรวจสอบสิทธิ์ ไม่ใช่การอนุญาต cross-origin

9. ถ้า API ไม่รองรับ CORS จะใช้วิธีแก้ไขไหน (JSONP, Proxy, etc.)

- ตอบ : ถ้า API ไม่รองรับ CORS วิธีแก้คือใช้ Proxy server หรือเทคนิคเก่าอย่าง JSONP เพื่อให้ client เข้าถึงข้อมูลได้

10. CORS error ส่งผลต่อการแสดงผลหน้าเว็บหรือไม่ (Network tab vs Console)

- ตอบ : CORS error จะทำให้หน้าเว็บไม่สามารถแสดงข้อมูลที่ดึงมาได้ แต่ใน Network tab อาจเห็น request ถูกส่งไปแล้ว ส่วน Console จะโชว์ข้อความ error ชัดเจน

## 🎯 เรื่อง 5: DOM Manipulation และ Events

1. Element `#app` ใช้เก็บอะไร และทำไมถึงเลือก `innerHTML` แทน `textContent`

- ตอบ : Element #app ใช้เป็น container หลักสำหรับ render เนื้อหาในแต่ละหน้า การเลือกใช้ innerHTML แทน textContent เพราะต้องการใส่โครงสร้าง HTML ทั้งหมด ไม่ใช่แค่ข้อความธรรมดา

2. Event listener `click` ที่ใช้ใน movie card ทำหน้าที่อะไร (ดูจาก `onclick`)

- ตอบ : Event listener click ที่ใช้ใน movie card ทำหน้าที่เปลี่ยน hash ของ URL ไปยัง route #/movie/:id เพื่อให้ router โหลดรายละเอียดหนังเรื่องนั้น

3. Movie card ใช้ onclick attribute แทน addEventListener ทำไม

- ตอบ : Movie card ใช้ onclick attribute แทน addEventListener เพราะเป็นวิธีที่ง่ายและตรงไปตรงมาในการผูก event กับ element โดยตรง แม้ว่า addEventListener จะยืดหยุ่นกว่า

4. ฟังก์ชัน `handleSearch()` จะถูกเรียกใช้เมื่อเกิดเหตุการณ์ใด

- ตอบ : ฟังก์ชัน handleSearch() จะถูกเรียกเมื่อผู้ใช้กดปุ่มค้นหา หรือพิมพ์ในช่อง input แล้วกด Enter/onkeyup ทำให้ระบบไปดึงข้อมูลจาก API

5. ฟังก์ชัน `handleSearch()` จำเป็นต้องเรียก `Views.loading()` ทำไม

- ตอบ : การเรียก Views.loading() จำเป็นเพื่อแสดงสถานะกำลังโหลดให้ผู้ใช้เห็นระหว่างรอผลลัพธ์จาก API ป้องกันความรู้สึกว่าเว็บค้าง

6. การแสดงรูปภาพในการ์ด ถ้าไม่มี poster จะแสดงอะไรแทน

- ตอบ : การแสดงรูปภาพในการ์ดจะใช้โปสเตอร์จาก API แต่ถ้าไม่มีหรือเป็น "N/A" จะใช้รูปแทนที่จาก placeholder URL

## ⏳ เรื่อง 6: Async/Await และ Promise

1. `async function` คืออะไร และมีข้อดีกว่า `.then()` หรือไม่

- ตอบ : async function คือฟังก์ชันที่คืนค่าเป็น Promise โดยอัตโนมัติ ข้อดีคือเขียนโค้ดอ่านง่ายกว่า .then() เพราะใช้โครงสร้างเหมือน synchronous code

2. Keyword `await` จะทำให้โปรแกรมทำอะไร (halts execution หรือไม่)

- ตอบ : Keyword await จะหยุดการทำงานภายใน async function ชั่วคราวจนกว่าจะได้ผลลัพธ์จาก Promise แต่ไม่ได้หยุดทั้งโปรแกรม เพียงแต่รอผลลัพธ์เฉพาะบรรทัดนั้น

3. ฟังก์ชัน `API.getMovies()` เรียกใช้ `async` ทำไม

- ตอบ : ฟังก์ชัน API.getMovies() ใช้ async เพราะต้องรอผลลัพธ์จาก fetch() ซึ่งเป็น asynchronous operation ทำให้สามารถใช้ await ได้สะดวก

4. ฟังก์ชัน `navigate()` ใช้ `async` และ `await` อย่างไร

- ตอบ : ฟังก์ชัน navigate() ใช้ async เพื่อรอข้อมูลจาก API เช่น getMovies() หรือ getMovieById() ก่อนจะ render หน้าใหม่ ป้องกันการแสดงผลที่ยังไม่สมบูรณ์

5. Try-catch block ใน `API.getMovies()` จะจับ error ประเภทไหน

- ตอบ : Try-catch block ใน API.getMovies() จะจับ error ประเภท network error, fetch error หรือ JSON parsing error เพื่อไม่ให้โปรแกรม crash

6. ถ้า `await fetch()` fail จะเกิดอะไร (ให้อธิบาย error handling)

- ตอบ : ถ้า await fetch() fail จะ throw error และไปเข้าที่ catch block ซึ่งจะแสดงข้อความ error หรือคืนค่าเป็น array ว่างตามที่โค้ดกำหนด

7. Promise chain (`.then().catch()`) ส่วนไหนเหมือน async/await

- ตอบ : Promise chain .then().catch() ทำงานเหมือน async/await ตรงที่รอผลลัพธ์และจัดการ error แต่ async/await ทำให้โค้ดอ่านง่ายกว่าและเป็นลำดับขั้นชัดเจน

8. Callback function `handleSearch()` ข้างใน `onkeyup` เป็น promise หรือไม่

- ตอบ : Callback function handleSearch() ที่ถูกเรียกจาก onkeyup ไม่ใช่ Promise โดยตรง แต่ภายในมันใช้ async/await ทำให้การทำงานกับ API เป็น asynchronous

9. ฟังก์ชัน `navigate()` await ทั้ง `getMovies()`, `getMovieById()`, `searchMovies()` ทำไม

- ตอบ : ฟังก์ชัน navigate() ใช้ await กับ getMovies(), getMovieById(), searchMovies() เพราะต้องรอข้อมูลจาก API ก่อน render UI มิฉะนั้นจะได้หน้าเปล่าหรือ error

10. ถ้า API response ช้า user จะเห็นอะไรขณะรอ (ให้อธิบายจาก Views.loading())

- ตอบ : ถ้า API response ช้า หน้าเว็บจะแสดงข้อความจาก Views.loading() เช่น “⏳ กำลังโหลด...” เพื่อบอกผู้ใช้ว่าระบบกำลังทำงานอยู่ ไม่ใช่ค้าง

## 🔐 เรื่อง 10: Security และ Best Practices

1. API key ไว้ใน client-side code มีความเสี่ยงหรือไม่ เหตุใด

- ตอบ : การเก็บ API key ไว้ใน client-side code มีความเสี่ยง เพราะผู้ใช้สามารถเปิดดู source code หรือ network request แล้วเห็น key ได้ อาจถูกนำไปใช้โดยไม่ได้รับอนุญาต

2. วิธีแก้ไข security issue ของ API key คืออะไร

- ตอบ : วิธีแก้คือเก็บ API key ฝั่ง server แล้วให้ client เรียกผ่าน proxy หรือ environment variables เพื่อไม่ให้ key ถูกเปิดเผยใน browser

3. OMDB API มีการจำกัด rate limit หรือไม่ (ฟรี account ได้กี่ requests)

- ตอบ : OMDB API มีการจำกัด rate limit โดยบัญชีฟรีอนุญาตให้เรียกได้จำนวนจำกัดต่อวัน (เช่น 1,000 requests) เพื่อป้องกันการใช้งานเกินควร

4. Error handling ควรมี try-catch หรือเพียง `.catch()` ก็พอ

- ตอบ : Error handling ควรใช้ try-catch ร่วมกับ .catch() เพื่อครอบคลุมทั้ง synchronous และ asynchronous error ทำให้โค้ดปลอดภัยและไม่ crash

5. Input validation ของการค้นหา ควรตรวจสอบอะไรบ้าง

- ตอบ : Input validation ของการค้นหาควรตรวจสอบว่าไม่เป็นค่าว่าง, ไม่มี script injection และเป็นข้อความที่เหมาะสมสำหรับ query

6. Script tag order ใน HTML มีความสำคัญ (api.js ก่อน main.js) ทำไม

- ตอบ : Script tag order สำคัญ เช่น api.js ต้องโหลดก่อน main.js เพราะ main.js เรียกใช้ API object ถ้าโหลดผิดลำดับจะเกิด error

7. `console.error()` ใช้ในไหนบ้าง และควรแสดงให้ user เห็นหรือไม่

- ตอบ : console.error() ใช้สำหรับ log ข้อผิดพลาดเพื่อ debug ไม่ควรแสดงให้ user เห็นตรง ๆ แต่ควรแสดงข้อความ error ที่เป็นมิตรแทน

8. Sensitive data (API key) ไม่ควรโปรแกรมใน source code แต่ดึงจากไหน

- ตอบ : Sensitive data เช่น API key ไม่ควร hardcode ใน source code แต่ควรดึงจาก environment variables หรือ config ที่ฝั่ง server

9. HTTPS จำเป็นสำหรับคำขอ OMDB API หรือไม่

- ตอบ : HTTPS จำเป็นสำหรับการเรียก OMDB API เพราะช่วยเข้ารหัสข้อมูล ป้องกันการดักจับหรือแก้ไข request ระหว่างทาง

10. Browser security policy จะบล็อก request ไปยัง HTTP endpoint หรือไม่ (OMDB uses HTTPS)

- ตอบ : Browser security policy จะไม่อนุญาตให้ request ไปยัง HTTP endpoint ถ้าเว็บโหลดผ่าน HTTPS ดังนั้น OMDB ใช้ HTTPS เพื่อให้ปลอดภัยและเข้ากันได้กับ browser
